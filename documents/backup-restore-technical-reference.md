# Tally Backup, Restore & Factory Reset — Technical Reference

## Architecture Overview

The backup/restore system is designed for **safety, atomicity, and data integrity**. All operations are designed to leave the database in a consistent state, even if interrupted mid-operation.

```
┌─────────────┐
│  Backup     │ → TAR.GZ (JSON + SQL dump + manifest)
├─────────────┤
│  Restore    │ ← Validates checksums → Atomic transaction
├─────────────┤
│  Reset      │ → FK-safe deletes → Re-seeds categories/tags/views → Sample data
└─────────────┘
```

## Backup Architecture

### Backup Process (Backend: `services/backup.py`)

```
1. connect_to_sqlite()
   ↓
2. execute: SELECT * FROM each_table_in_order
   ↓
3. serialize to JSON (handle date/datetime/JSON columns)
   ↓
4. generate_sql_dump() via asyncio.to_thread(sqlite3.iterdump)
   ↓
5. compute_sha256_checksums() for each file
   ↓
6. create_manifest.json with checksums + metadata
   ↓
7. tar_gzip(json_dir + sql_dir + manifest) → ~/.tally/backups/
   ↓
8. return {filename, filesize_bytes, timestamp}
```

### Export Table Order

Tables are exported in **parent-before-child** order (foreign key dependency order) for clarity:

```
1. accounts           (no FK dependencies)
2. categories         (self-ref parent_id, but no external deps)
3. tags               (no FK dependencies)
4. schema_mappings    (FK: account_id)
5. import_batches     (FK: account_id)
6. transactions       (FK: account_id, import_batch_id, categories, self-ref transfer_pair_id)
7. transaction_tags   (FK: transaction_id, tag_id — composite PK join table)
8. correction_history (FK: transaction_id, category_id refs)
9. monthly_snapshots  (no FK dependencies from above)
10. snapshot_items    (FK: snapshot_id, account_id)
11. net_worth_views   (no FK)
12. recurring_bills   (FK: category_id)
13. settings          (string PK, not integer)
14. stock_holdings    (no FK)
15. stock_price_history (no FK)
```

```
│   ├── "version": "1"

│     }
│
├── json/
│   ├── accounts.json              (Array of account objects)
│   ├── categories.json

│   ├── transactions.json
```

### JSON Serialization Format

Each JSON file is a plain array of objects:

```json
[
  {
    "id": 1,
    "name": "Chase Checking",
    "type": "checking",
    "institution": "Chase",
    "account_number_masked": null,
    "currency": "USD",
    "color": null,
    "is_active": true,
    "created_at": "2026-05-20T12:30:00+00:00"  // ISO 8601 format
  },
  {
    "id": 2,
    "name": "Ally Savings",
    ...
  }
]
```

#### Special Handling for Column Types

| Column Type | JSON Representation | Notes |
|---|---|---|
| `Integer` | `123` | As-is |
| `String` | `"text"` | JSON string |
| `Float` | `123.45` | As-is |
| `Boolean` | `true`/`false` | As-is |
| `Date` | `"2026-05-20"` | ISO 8601 YYYY-MM-DD |
| `DateTime` | `"2026-05-20T12:30:00+00:00"` | ISO 8601 with timezone |
| `JSON` (dict/list) | `{"key": "value"}` / `[1, 2, 3]` | Nested JSON object/array |
| `NULL` | `null` | JSON null |

The `settings` table has a string primary key (`key`) instead of integer `id`:


The `transaction_tags` join table has a **composite primary key** and no surrogate `id`:

```json

  },
  {
    "transaction_id": 1,
    "tag_id": 7
  },
  {
    "transaction_id": 2,
    "tag_id": 5
  }
]
```

### SQL Dump Format

The `sql/tally_dump.sql` file is the output of SQLite's `.dump` command:

```sql
-- SQLite3 (auto-generated by sqlite3.iterdump())
BEGIN TRANSACTION;

CREATE TABLE accounts (
  id INTEGER PRIMARY KEY,
  name VARCHAR(200) NOT NULL,
  ...
);

CREATE TABLE categories (
  id INTEGER PRIMARY KEY,
  name VARCHAR(200) NOT NULL,
  parent_id INTEGER,
  ...
);

INSERT INTO accounts VALUES(1, 'Chase Checking', 'checking', 'Chase', NULL, 'USD', NULL, 1, '2026-05-20 12:30:00');
INSERT INTO accounts VALUES(2, 'Ally Savings', 'savings', 'Ally', NULL, 'USD', NULL, 1, '2026-05-20 12:30:00');

...

COMMIT;
```

This is a complete, runnable SQL script that fully recreates the database schema and data.

### Checksum Verification

Each backup file is verified using SHA-256:

```python
# During backup creation
sha256(file_contents).hexdigest() → "abc123def456..." 
stored in manifest.json as "sha256:abc123def456..."

# During restore
sha256(extracted_file).hexdigest() compared to stored checksum
if mismatch → ValueError("Checksum mismatch for {path_key}")
```

This ensures:
- Files weren't corrupted during download
- Files weren't tampered with
- Archive integrity is guaranteed before restore begins

## Restore Architecture

### Restore Entry Point (Backend: `routers/admin.py`)

```
POST /api/admin/restore (UploadFile)
  ↓
restore_from_upload(db, file_bytes)
  ↓
1. open_tar_gz(file_bytes)
2. read_manifest_json()
3. verify_all_checksums()
   └─ if any checksum mismatch → raise ValueError
4. detect_format (has JSON? has SQL?)
   ├─ if JSON → restore_from_json(db, table_data)
   └─ if SQL → restore_from_sql(sql_text)
5. return {"ok": True, "method": "json"|"sql"}
```

### JSON Restore Path (Atomic, Recommended)

**Located in:** `services/restore.py::_restore_from_json(db, table_data)`

```
async with db.begin():          # Start transaction
  ↓
PRAGMA foreign_keys = OFF       # Disable FK constraints
  ↓
DELETE FROM all_tables (FK-safe order)
  1. correction_history
  2. transaction_tags
  3. transactions
  4. import_batches
  5. recurring_bills
  6. schema_mappings
  7. snapshot_items
  8. monthly_snapshots
  9. stock_price_history
  10. stock_holdings
  11. net_worth_views
  12. tags
  13. categories
  14. accounts
  15. settings
  ↓
INSERT all rows (reverse FK order)
  15. accounts
  14. categories
  13. tags
  12. net_worth_views
  11. stock_holdings
  10. stock_price_history
  9. monthly_snapshots
  8. snapshot_items
  7. schema_mappings
  6. recurring_bills
  5. transactions
  4. import_batches
  3. transaction_tags
  2. correction_history
  1. settings
  
  Special handling:
  - JSON columns (dicts/lists) are re-serialized via json.dumps()
  - Dates serialized as ISO strings are stored as-is
  - Composite PKs (transaction_tags) use INSERT OR REPLACE
  - String PKs (settings) use INSERT OR REPLACE
  ↓
PRAGMA foreign_keys = ON       # Re-enable FK constraints
  ↓
[implicit COMMIT on exit]      # Atomicity: all-or-nothing
```

**Atomicity Guarantees:**
- If ANY insert fails → entire transaction rolls back
- Database returns to pre-restore state
- No partial data corruption possible
- User is notified of the specific error

**Why Atomic?**
This design prevents "stranded" data:
- If restore dies midway, you don't have transactions without accounts
- If a checksum fails mid-restore, you don't have half-restored data
- Either the restore completes fully, or nothing changes

### SQL Restore Path (Fallback)

**Located in:** `services/restore.py::_restore_from_sql(sql_text)`

Used when the backup archive lacks JSON files (rare, fallback path):

```
1. await engine.dispose()           # Close all pooled SQLAlchemy connections
   ↓
2. shutil.copy2(db_path, backup)   # Safety: backup the current DB file
   ↓
3. await asyncio.to_thread(_sync_apply_sql_dump, db_path, sql_text)
   └─ conn = sqlite3.connect(db_path)
   └─ conn.executescript(sql_text)    # Run full SQL dump in one transaction
   └─ conn.close()
   ↓
4. unlink(backup)                   # Delete safety backup (restore succeeded)
   ↓
5. Engine auto-reconnects on next API request
```

**Error Handling:**
```python
try:
    # ... restore attempt
except Exception:
    if backup.exists():
        shutil.copy2(backup, db_path)  # Restore original file
        backup.unlink()
    raise  # Re-raise error to caller
```

If SQL restore fails, the original database file is restored.

## Factory Reset Architecture

### Delete Order (FK-Safe)

The following order ensures no foreign key constraint violations:

**Child Tables First** (tables that reference others):
```
1. correction_history      ← FK: transaction_id, category_id
2. transaction_tags        ← FK: transaction_id, tag_id (composite PK)
3. transactions            ← FK: account_id, import_batch_id, category_id, transfer_pair_id (self)
4. import_batches          ← FK: account_id
5. recurring_bills         ← FK: category_id
6. schema_mappings         ← FK: account_id
7. snapshot_items          ← FK: snapshot_id, account_id
8. monthly_snapshots       ← (no FK to others; children deleted first)
9. stock_price_history     ← (no FK)
10. stock_holdings         ← (no FK)
11. net_worth_views        ← (no FK)
12. tags                   ← (no FK from above; used by transaction_tags)
13. categories             ← (no FK from above; used by transactions; self-ref parent_id)
14. accounts               ← (no FK from above; used by many)
15. settings               ← (no FK; just key-value store)
```

**Note on Self-References:**
- `categories.parent_id → categories.id` — children before parents
  - With FK off, deletion order doesn't matter; all are deleted regardless
  - But this order makes the intent clear: "delete children first"
  
- `transactions.transfer_pair_id → transactions.id` — self-reference
  - Both rows deleted in same step (order doesn't matter with FK off)

### Reset Process (Backend: `services/factory_reset.py`)

```
async def factory_reset(db):
  ↓
  DELETE all 15 tables in FK-safe order
    └─ within async with db.begin(): for atomicity
  ↓
  await _seed_categories(db)
    ├─ INSERT 15 parent categories (Income, Housing, Food, ...)
    └─ INSERT ~40 subcategories (Salary, Groceries, Rent, ...)
  ↓
  await _seed_tags(db)
    └─ INSERT 6 tags (Nick, Emma, Family, Work, Cat Stuff, Subscriptions)
  ↓
  await _seed_net_worth_views(db)
    └─ INSERT 7 pre-built views (Total NW, Liquid Assets, Real Estate, ...)
  ↓
  await _seed_sample_data(db)
    ├─ INSERT 6 sample accounts
    ├─ INSERT 6 stock holdings (VTI, VXUS, BND, AAPL, MSFT, SPY)
    ├─ INSERT 1 monthly_snapshot dated 2026-05-19
    ├─ INSERT 9 snapshot_items (6 assets + 3 liabilities)
    └─ INSERT 6 transactions (May 2-19, 2026)
  ↓
  return {"ok": True}
```

## Edge Cases & Special Scenarios

### Composite Primary Key: `transaction_tags`

**Problem:** `transaction_tags` has a composite PK (no surrogate `id` column):
```sql
CREATE TABLE transaction_tags (
  transaction_id INTEGER NOT NULL,
  tag_id INTEGER NOT NULL,
  PRIMARY KEY (transaction_id, tag_id),
  FOREIGN KEY (transaction_id) REFERENCES transactions(id),
  FOREIGN KEY (tag_id) REFERENCES tags(id)
)
```

**JSON Representation:**
```json
[
  {"transaction_id": 1, "tag_id": 5},
  {"transaction_id": 1, "tag_id": 7}
]
```

**Restore SQL:**
```python
INSERT OR REPLACE INTO transaction_tags (transaction_id, tag_id) 
VALUES (:transaction_id, :tag_id)
```

**Why INSERT OR REPLACE?**
- Multiple rows with same (transaction_id, tag_id) would be duplicates
- `INSERT OR REPLACE` silently skips duplicates (idempotent)
- Safer than `INSERT` which would raise a constraint error

### String Primary Key: `settings`

**Problem:** `settings` uses a string as primary key:
```sql
CREATE TABLE settings (
  key STRING PRIMARY KEY,
  value JSON NOT NULL
)
```

**JSON Representation:**
```json
[
  {"key": "display_config", "value": {...}},
  {"key": "another_setting", "value": "..."}
]
```

**Restore SQL:**
```python
INSERT OR REPLACE INTO settings (key, value) 
VALUES (:key, :value_json_string)
```

**Why INSERT OR REPLACE?**
- `INSERT` would fail if key already exists
- `INSERT OR REPLACE` acts as "upsert"
- Handles duplicate keys gracefully

### JSON Column Values

**Problem:** Columns that store JSON (dicts/lists) must be serialized to strings for SQL:

Example: `net_worth_views.definition` is JSON:
```python
# Python object (from SELECT)
{"include_types": ["all"], "exclude_types": [], ...}

# Must be re-serialized for SQL
INSERT INTO net_worth_views (definition) VALUES ('{"include_types": ["all"], ...}')
```

**Solution in restore code:**
```python
serialized = {}
for k, v in row.items():
    if isinstance(v, (dict, list)):
        serialized[k] = json.dumps(v)  # Re-serialize
    else:
        serialized[k] = v  # Keep as-is
```

### Self-Referential Foreign Keys

**Problem 1: `categories.parent_id → categories.id`**
- If categories are inserted out of order, parent might not exist yet
- Solution: Insert with `PRAGMA foreign_keys = OFF`

**Problem 2: `transactions.transfer_pair_id → transactions.id`**
- Transaction A might reference Transaction B before B is inserted
- Solution: Insert with `PRAGMA foreign_keys = OFF`

With FK enforcement disabled during restore, order doesn't matter. Re-enabled after all inserts.

### asyncio.to_thread for Synchronous sqlite3

**Problem:** SQLite's `iterdump()` is synchronous and would block the async event loop.

**Solution:**
```python
sql_dump = await asyncio.to_thread(_sync_sql_dump, settings.sqlite_path)

def _sync_sql_dump(db_path: str) -> str:
    conn = sqlite3.connect(db_path)
    try:
        return "\n".join(conn.iterdump())
    finally:
        conn.close()
```

This runs the blocking code in a thread pool, freeing the event loop to handle other requests.

## CLI Commands (Manual Recovery)

### Manual Restore via SQLite CLI

If the app is unavailable, restore using `sqlite3` directly:

```bash
# Navigate to backup directory
cd ~/.tally/backups

# Extract backup
tar -xzf tally_backup_20260520_143022.tar.gz

# Option 1: Restore from SQL dump (fastest)
sqlite3 ~/.tally/tally.db < sql/tally_dump.sql

# Option 2: Restore from JSON (requires custom script — see below)
python3 restore_from_json.py json/ ~/.tally/tally.db
```

### Manual JSON Restore Script

For advanced users, here's pseudocode to restore from JSON:

```python
import json
import sqlite3

def restore_from_json_files(json_dir, db_path):
    conn = sqlite3.connect(db_path)
    conn.execute("PRAGMA foreign_keys = OFF")
    
    # Delete all tables (in FK order)
    DELETE_ORDER = [...]
    for table in DELETE_ORDER:
        conn.execute(f"DELETE FROM {table}")
    
    # Re-insert (reverse FK order)
    for table in reversed(DELETE_ORDER):
        with open(f"{json_dir}/{table}.json") as f:
            rows = json.load(f)
        for row in rows:
            cols = ", ".join(row.keys())
            placeholders = ", ".join(f":{k}" for k in row.keys())
            conn.execute(
                f"INSERT INTO {table} ({cols}) VALUES ({placeholders})",
                row
            )
    
    conn.execute("PRAGMA foreign_keys = ON")
    conn.commit()
    conn.close()
```

## Troubleshooting

### Checksum Mismatch During Restore

**Symptom:**
```
ValueError: Checksum mismatch for json/accounts.json
```

**Causes:**
1. File corrupted during download
2. File manually edited after backup
3. Disk error during backup creation

**Solution:**
```bash
# Try restoring from SQL dump instead (if present)
# Download backup again and retry
# Or use a different backup if available
```

### Restore Fails Mid-Operation

**Symptom:**
```
500 Internal Server Error: Restore failed
```

**What happens:**
- Transaction automatically rolls back
- Original database is unchanged
- Try restore again with same backup

**Debug:**
- Check backup file integrity (checksums pass?)
- Try extracting the tar.gz manually
- Check disk space on `~/.tally/`
- Look for error details in app logs

### Backup File is Corrupted

| Archive Size | Approximate Time |
|---|---|

### Disk Space Requirements

- **Backup creation:** ~3x database size (temp working space)
- **Restore operation:** ~2x archive size
- **Backup storage:** ~20-30% of database size (compressed)

## Security Considerations

### Data Encryption

**Current:** Backups are NOT encrypted.

**Recommendation:** Store backups on encrypted devices or use encrypted cloud storage (Google Drive encryption, Dropbox encryption, etc.).

### Access Control

**Current:** No backup-level access control (any user can download/restore).

**Recommendation:** Restrict backup directory permissions if running on shared systems:
```bash
chmod 700 ~/.tally/backups
```

### Deletion Protection

**Current:** Backups can be deleted freely.

**Recommendation:** 
1. Store backups in read-only locations
2. Use cloud storage with version history (Google Drive, Dropbox)
3. Keep offline copies for critical backups
